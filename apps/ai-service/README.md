# @support-saas/ai-service

FastAPI microservice: RAG-backed knowledge base retrieval, AI-suggested
replies, and ticket summarization/sentiment. Called by the Node API as an
internal service — never exposed directly to the browser.

## Architecture

- **LlamaIndex** owns chunking (`SentenceSplitter`), embedding, and org-scoped
  retrieval against **Qdrant**. Every point is tagged with `org_id` in its
  payload and every query applies an `org_id` filter — tenants share one
  Qdrant collection but can never retrieve each other's articles.
- **LangChain** owns prompting and generation: a structured-output chain for
  summarize/sentiment, and a RAG-synthesis prompt for suggested replies that
  grounds the answer in the chunks LlamaIndex retrieved.
- **LLM/embeddings**: Google Gemini, via `langchain-google-genai`. `app/llm.py`
  is the only place that constructs the chat/embedding clients — get a free
  API key at https://aistudio.google.com/apikey and set `GOOGLE_API_KEY` in
  `.env`. Default models: `gemini-2.5-flash` for chat, `models/text-embedding-004`
  (768 dimensions — matches `QDRANT_VECTOR_SIZE` below) for embeddings.
- **Why HTTP, not just Kafka**: the Kafka topics for AI events already exist
  in `@support-saas/shared-types` for future analytics/audit consumers, but
  the user-facing suggested-reply flow needs a synchronous answer while an
  agent is looking at a ticket, so the Node API calls this service directly
  over HTTP and returns the result inline.

## Endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | Liveness + which providers are configured |
| POST | `/ai/index/article` | Chunk + embed a knowledge article into Qdrant |
| DELETE | `/ai/index/article/{id}` | Remove all chunks for an article |
| POST | `/ai/summarize` | Ticket transcript → summary + sentiment |
| POST | `/ai/suggest-reply` | RAG: retrieve relevant articles, draft a grounded reply |

Full request/response schemas: run the service and see `/docs` (Swagger UI,
auto-generated by FastAPI from `app/schemas.py`).

## Setup

```bash
cd apps/ai-service
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # fill in GOOGLE_API_KEY (https://aistudio.google.com/apikey)
uvicorn app.main:app --reload --port 8000
```

## Testing

```bash
pip install -r requirements.txt   # includes pytest
python -m pytest -v               # 15 tests, ~2s, no external services required
```

`tests/conftest.py` provides a `fake_vector_store` fixture (in-memory Qdrant
+ a deterministic hash-based embedder) and a `client` fixture (FastAPI
`TestClient`), so the whole suite runs offline with no Gemini API key, no Qdrant
server, and no network access — exactly what CI does (`.github/workflows/ci.yml`).

## Verified

This service's core logic was tested end-to-end in the build sandbox using
the same in-memory Qdrant + deterministic fake embedder now checked in as
`tests/`:

- **Chunking + indexing**: articles are split and embedded correctly.
- **Org isolation**: a query scoped to `org_a` never returns `org_b`'s
  articles, and vice versa — verified with an explicit cross-tenant leak
  assertion.
- **Deletion**: re-indexing and deleting an article correctly removes its
  chunks from subsequent retrievals (`test_delete_article_before_any_index_does_not_crash`
  is a regression test for a real crash caught during this build — see below).
- **suggest_reply retrieval + synthesis**: with a mocked chat model, the
  full retrieve → build-context → synthesize pipeline was exercised and
  produced a reply citing the correct source article.
- **Graceful no-match fallback**: a query against an org with no indexed
  articles returns confidence `0.0` and an honest "a human should draft
  this" message instead of hallucinating.
- **Live route registration and error handling**: the FastAPI app was
  started for real; `/health` returns 200, and calls that need Gemini or
  Qdrant (unavailable in the sandbox) fail with a clean `502` and a
  descriptive message rather than crashing the process.

An actual Gemini API call and a real Qdrant server were **not** reachable from
the build sandbox (network egress there is limited to package registries),
so wire up real credentials/infra and re-run before depending on this in
production — but the integration code paths themselves are exercised and
correct, not just written.

## Node API integration (Phase 5 wiring)

- `POST /api/knowledge` (Node) → indexes into this service on create/update
- `POST /api/tickets/:id/ai-suggest` (Node) → proxies to `/ai/suggest-reply`
- Ticket creation with an initial customer message best-effort calls
  `/ai/summarize` to populate `ticket.aiSummary` / `ticket.aiSentiment`
